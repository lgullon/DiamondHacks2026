This is a fantastic first hackathon idea — it's practical, accessible, and has a clear demo path. Let me break down everything you need.

The Big Picture Architecture
Before splitting work, understand how the pieces connect:
[Chrome Extension (Plasmo + React)]
        ↕ CDP (reads form fields)
        ↕ HTTP (talks to your backend)
[FastAPI Backend]
        ↕ Anthropic API (understands + simplifies questions)
        ↕ Whisper/ElevenLabs (voice in/out)
        ↕ Browser Use CLI (fills the form)
The extension is the face. The backend is the brain. Browser Use is the hands.

Work Split
Person A — Extension ("The Face") and Person B — Backend ("The Brain") can work almost entirely in parallel after the first hour.
Person A: Extension

Set up Plasmo project, get a sidebar/popup rendering
Build the mascot UI (animated character in corner)
Implement CDP calls to read form fields from the page
Record audio → send to backend → play back audio response
Trigger backend to fill fields via browser use

Person B: Backend

Set up FastAPI with all routes stubbed out
Integrate Anthropic API to simplify form questions + handle conversation
Integrate Whisper for speech-to-text
Integrate ElevenLabs for text-to-speech
Integrate Browser Use to fill fields


The API Routes (Your Command Blocks)
pythonPOST /analyze-form        # receives form fields JSON, returns simplified questions
POST /voice-input         # receives audio blob, returns transcribed text  
POST /chat                # receives user message + context, returns AI response + fill instruction
POST /fill-field          # triggers browser use to fill a specific field
POST /speak               # converts text to speech, returns audio

The Conversation Flow (State Machine)
1. User clicks mascot → extension reads all form fields via CDP
2. Extension sends fields to /analyze-form → backend returns simplified Q&A list
3. For each field:
   a. Backend calls /speak with simplified question → plays audio to user
   b. User speaks → extension sends audio to /voice-input → transcription returned
   c. Transcription sent to /chat → Claude determines: answer or follow-up question?
   d. If follow-up: loop back to (b)
   e. If answer: call /fill-field → browser use fills it → move to next field
4. User reviews → clicks submit

Your Demo Form Strategy
Build a simple HTML form with these field types to show range:

A text input (name, date of birth)
A <select> dropdown (insurance type, language preference)
Radio buttons (yes/no questions like "Do you have allergies?")
A textarea (describe your symptoms)

Host it as a simple index.html on localhost or GitHub Pages. You control it completely, so you can make sure Browser Use fills it perfectly every time.

Vibe Coding Tips for First Hackathons

Agree on your API contract in Hour 1 and don't change field names. Person A mocks Person B's responses and vice versa — you stay unblocked.
Get something end-to-end ugly working by Hour 8. No mascot animation, no multilingual — just: reads field → asks question → fills it. Then layer on features.
The demo form is your safety net. If browser use is flaky on random sites, that's fine. Your curated form should work perfectly every time.
Stub aggressively. If ElevenLabs is taking too long to integrate, have the backend return text and use the browser's built-in speechSynthesis API as a fallback.
One laptop runs the demo. Decide now whose laptop it is, get everything running on it by Hour 20.
Name your mascot. It makes the demo memorable. Judges remember "FormBuddy" or "Lingo" way longer than "our extension."


The One Risk to Manage
Browser Use needs to control the browser — but you're also running a Chrome extension. These can conflict. The safest pattern: Browser Use runs in a separate Chrome profile or via Playwright in the backend, while your extension just reads fields via CDP and triggers the backend. Don't have both trying to control the same tab simultaneously.

Concretely, here's your API contract:
Both of you open a shared doc (Google Doc, Notion, whatever) and write something like this on Day 1:
POST /analyze-form
json// Person A sends:
{
  "fields": [
    { "id": "dob", "type": "text", "label": "Date of Birth" },
    { "id": "insurance", "type": "select", "label": "Insurance Type", "options": ["HMO", "PPO", "None"] }
  ],
  "language": "es"
}

// Person B returns:
{
  "questions": [
    { "id": "dob", "simplified": "¿Cuándo naciste? (ejemplo: 15 de marzo, 1990)" },
    { "id": "insurance", "simplified": "¿Qué tipo de seguro médico tienes?" }
  ]
}
POST /chat
json// Person A sends:
{ "field_id": "dob", "user_message": "what does date of birth mean?", "language": "es" }

// Person B returns:
{ "reply": "Es la fecha en que naciste.", "ready_to_fill": false, "fill_value": null }
// OR when they give a real answer:
{ "reply": "Perfecto, voy a escribir eso.", "ready_to_fill": true, "fill_value": "03/15/1990" }

Once you both agree on this, Person A can just hardcode fake responses that match the shape, and build the whole UI/audio flow. Person B builds the real logic independently. When you connect at Hour 8, it just works — because the "plug shape" already matches.
The most common first-hackathon mistake is skipping this and then spending 3 hours debugging because one person returns user_text and the other is reading transcription. Agree on field names first!
